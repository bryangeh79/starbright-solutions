# 07_minimum_rollback_plan

## 1. 回滚负责人

- 角色：项目维护人 / 部署执行人
- 说明：当前未在工作区中找到具体姓名或平台账号信息，因此先以角色定义回滚责任。

## 2. 回滚介质

- 推荐依赖项：代码仓上一稳定版本
- 推荐依赖项：上一版构建产物
- 推荐依赖项：手工备份包
- 推荐依赖项：托管平台上一部署版本
- 当前原则：若真实部署平台尚未明确，优先使用“上一稳定版本 + 上一版构建产物”作为最小回滚依据。

## 3. 回滚触发条件

满足以下任一情况时，应考虑回滚：

- 上线后核心页面无法访问
- 表单或联系入口异常
- 核心路由失效
- 严重样式错乱影响交付
- SEO 或基础渲染出现重大异常

## 4. 回滚执行方法

最小可执行步骤：

1. 停止当前错误版本继续发布或继续扩散
2. 切回上一稳定版本，或重新部署上一稳定构建产物
3. 验证六个核心页面可访问
4. 验证联系入口与表单链路正常
5. 验证基础 SEO 元素与首屏渲染无重大异常
6. 恢复对外访问

## 5. 回滚后确认步骤

回滚完成后，至少确认以下项目：

- `/` 可访问
- `/solutions` 可访问
- `/industries` 可访问
- `/about` 可访问
- `/faq` 可访问
- `/contact` 可访问
- 联系入口正常
- 无重大前台阻断

## 6. 当前缺口说明

- 当前工作区未提供真实部署平台信息
- 当前工作区未提供具体发布账号、仓库分支或构建产物路径
- 因此本说明采取“最小通用回滚原则”，用于先闭合上线准备中的回滚方案要求
